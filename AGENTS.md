# Engineering Skills & Workflow Standards (chatops)

This document defines expectations for contributors
to the chatops Go project.

---

## Project Domain Skills

- Comfortable with chat platform integrations (webhooks, bot APIs, event streams).
- Experienced with Go for long-running services and CLI tools.
- Familiarity with operational tooling patterns (command routing, auditing, access control).

---

## Go Engineering Standards

- Follow idiomatic Go style; run `gofmt`/`goimports` on changes.
- Use [kong](https://github.com/alecthomas/kong) for CLI parsing: each subcommand lives in `cmd/<name>/` as a `Cmd` struct with a `Run()` method, wired into the `cli` struct in `main.go` (same layout as [parquet-tools](https://github.com/hangxie/parquet-tools)).
- Handle errors explicitly and provide useful feedback.
- Keep code comments short but clear: state intent or the non-obvious "why" concisely, and do not restate what the code already says.
- Use `context.Context` for cancellable operations.
- Avoid leaking goroutines or unnecessary memory allocations.
- Non-test Go source files are preferred to be under 300 LOC and must never exceed 500 LOC; split larger files into cohesive units. Test files may exceed this limit when table coverage or fixtures make that clearer than splitting.

---

## Testing & TDD Expectations

- Each source file must have a one-to-one mapping with a corresponding test file (e.g., `foo.go` ↔ `foo_test.go`).
- Tests must cover both typical and edge-case scenarios.
- Use [testify](https://github.com/stretchr/testify) (`require`) for assertions.
- Use **table-driven tests** for core logic.
- Follow **TDD principles**: design tests before implementation, and ensure tests fail before implementing the feature.

---

## Documentation Quality

- Commands and flags must have clear help text.
- Output formats (plain text, JSON) should be consistent and stable.
- Documentation (`README.md`) must reflect all behavioral changes.
- User examples should be runnable and tested.
- Markdown prose must not use hard line breaks: keep each paragraph and list item on a single line and let renderers wrap. Code blocks and tables are exempt. Markdown files must not contain tabs; indent Go code inside code blocks with 4 spaces.

---

## Contribution Workflow Norms

- **Always commit after completing a task.** Once `make all` passes, create a commit immediately — do not wait to be asked.
- **Sign every commit.** Commits must carry a valid signature (`git commit -S`, or configure `commit.gpgsign true`); unsigned commits are rejected.
- **Keep the number of commits to a minimum.** Group related work into cohesive, logically-scoped commits and fold fixups into the commit they belong to (amend or rebase) rather than accumulating incremental or corrective commits.
- Commit messages should follow **Conventional Commits**.
- **Do not** add AI/agent attribution in repo artifacts. Commit messages, PRs, issues, and documents must not mention AI assistants, agents, or generation/provenance metadata such as `Co-Authored-By`, `Generated by ...`, `Created with ...`, or similar.
- Avoid breaking backward compatibility without clear migration notes.

---

## Quality Gates

- Code must pass formatting, linting, testing, and build checks (`make all`) before merging.
- New features and bug fixes must keep overall test coverage at or above the `main` branch. If a change would lower coverage, get explicit approval before merging and note the gap.
- Refactoring and cleanup must preserve behavior and pass validation.

---

## Task Tracking Process

When working on improvements or fixes from a review:

1. **Prepare `TODO.md`** - Create or update `TODO.md` with numbered, categorized items. Use checkboxes (`- [ ]`) to track completion.
2. **Make changes** - Implement the fix or improvement for the selected item(s).
3. **Validate** - Ensure `make all` passes (see Quality Gates).
4. **Commit** - Commit only the source code changes. Do **not** commit `TODO.md`.
5. **Mark complete** - Update `TODO.md` to check off the completed item(s) (`- [x]`).
